1. INTRODUCTION 
 
 
1.1      Purpose 
 
 
The purpose of this Software Design Document (SDD) is to provide a comprehensive design blueprint 
for the MySebenarnya System, commissioned by the Malaysian Communications and Multimedia 
Commission (MCMC). This document outlines the architectural structure, key components, and 
operational workflow required to build and maintain the system. 
 
This Software Design Document (SDD) serves as a foundational reference for the development of the 
Inquiry System, a digital solution designed to streamline the management of public inquiries. The 
document is intended for a wide audience including system developers, designers, stakeholders, 
administrators, and testers. It provides a structured overview of the system’s architecture, component 
interactions, and design rationale to ensure all parties involved have a clear understanding of the 
project’s goals and implementation pathway. 
 
This SDD has been prepared in alignment with best practices in software engineering and reflects the 
collaborative efforts of the project team to build a reliable, user-friendly, and transparent inquiry 
management platform. Each module outlined in this document plays a vital role in the seamless 
operation of the system, from user registration to inquiry tracking and resolution. 
 
By following the design outlined in this document, the team aims to ensure the system is scalable, 
maintainable, and capable of meeting both current and future needs of the intended users and 
administrative bodies. 
 
 
 
1.2      System Identification  
 
 
This Software Design Document (SDD) presents the system architecture and design specifications for 
the AuthenticityHub System, an initiative by AuthenticityHub to modernize the way public inquiries 
are managed. The system empowers users to create and manage their accounts, submit official inquiry 
forms, and ensure those inquiries are directed to the correct agencies or authorities for further action. 
 
Through real-time progress tracking and structured workflows, AuthenticityHub aims to enhance 
clarity, build trust between the public and service providers, and ensure timely responses to pressing 
public matters. The system serves as a centralized platform focused on improving service efficiency, 
communication flow, and public accountability.

1.3      System Overview 
 
 
 
1.3.1: Manage User (User Registration, Login and Profile Management): This module is the 
gateway for users to access the AuthenticityHub platform. It allows users to create accounts, log in 
securely, and manage personal profiles such as email, password, and contact information. The system 
maintains authentication control and ensures that only legitimate users can submit inquiries. The public 
user can register a new account, log in with their credentials, and update their profile details at any 
time, establishing a secure and personalized experience for each user interacting with the system. 
MCMC acts as the system administrator, monitoring user registration activities and handling security-
related functions such as resetting passwords or deactivating suspicious accounts. Agencies do not 
have direct involvement in this module, but user information may be displayed to them when handling 
assigned inquiries. 
 
 
1.3.2: Manage Inquiry Form Submission: This module provides a structured digital form that users 
can use to submit inquiries, reports, or complaints. It captures detailed information related to the 
user's issue and may include attachments like photos or documents to support their claim. The public 
user accesses the inquiry form, fills in the required details, selects the issue category, and submits it 
through the platform. This enables them to raise issues in an official and trackable way. MCMC 
reviews submitted forms for completeness and relevance, verifies the accuracy of the information, 
and decides whether the inquiry is valid and ready to be forwarded for assignment. Agencies do not 
interact with this module directly but will receive the inquiry only after MCMC has approved and 
assigned it.

1.3.3: Inquiry Assignment (to Agencies): This module enables MCMC to assign submitted inquiries 
to the appropriate agency based on the nature, category, and scope of the issue. The goal is to ensure 
that each inquiry reaches the right party for resolution. The public user is not involved in the 
assignment process but will be notified once their inquiry is forwarded to a relevant agency for action. 
MCMC evaluates each verified inquiry and uses this module to assign it to the correct agency, typically 
based on issue type, location, or the agency’s area of responsibility. The agency receives the assigned 
inquiry along with all supporting data and is expected to acknowledge the assignment and begin 
working on the resolution. 
 
 
1.3.4: Inquiry Progress Tracking: This module tracks the lifecycle of each inquiry, from 
submission to resolution. It ensures transparency by updating the status and making it visible to all 
relevant parties, including status labels, timestamps, and action logs. The public user can log in to 
view the status of their inquiry, check progress updates, and read any agency responses, helping 
build trust by keeping the user informed. MCMC monitors the progress of all inquiries, ensures 
agencies are updating statuses, and may intervene if inquiries are delayed or left unresolved. 
Agencies use this module to update the status of each inquiry as they work on it, marking it as “In 
Progress”, “Resolved”, or “Closed”, and may add comments or feedback for the user.

The Entity-Relationship Diagram (ERD) illustrates a system designed to manage public inquiries and their resolution 
through administrative assignment and agency action. This diagram includes seven main entities: Administrator, PublicUser, 
Agency, Inquiry, AssignedInquiry, Report, and Inquiry, highlighting their relationships and attributes. 
 
1. Administrator Entity: 
This table stores information about system administrators who manage the inquiry system. It includes basic details 
like AdminID, AdminName, AdminEmail, AdminRole, and login credentials. Administrators are responsible for 
assigning inquiries to the right agencies and overseeing the resolution process. 
 
2. PublicUser Entity: 
This table contains information about citizens who submit inquiries to the system. It 
includes UserID, UserName, UserEmail, UserPhoneNum, and UserProfilePicture. Public users create inquiries when 
they have questions or complaints about communications and multimedia services. 
 
3. Agency Entity: 
This table stores details about government agencies that handle different types of inquiries. It 
includes AgencyID, AgencyName, AgencyEmail, AgencyType, and login credentials. Each agency specializes in 
resolving specific types of inquiries based on their expertise. 
 
4. Inquiry Entity: 
This is the main table that stores all inquiry details submitted by public users. It 
includes InquiryID, InquiryTitle, InquiryDescription, InquiryStatus, InquiryEvidence, and links to the user who 
submitted it. This table tracks each inquiry from submission to resolution. 
 
5. AssignedInquiry Entity: 
This table manages the assignment of inquiries to agencies by administrators. It connects AdminID, AgencyID, 
and InquiryID with an AssignedDate and AdminNote. This ensures each inquiry is properly assigned and tracked. 
 
 
Relationships: 
 
1. Public users submit inquiries through the system 
2. Administrators review and assign inquiries to appropriate agencies 
3. Agencies work on resolving the assigned inquiries 
4. The system tracks all activities and maintains records for accountability

3. GENERAL ARCHITECTURE 
 
3.1 Architectural Pattern: MVC 
 
The system utilizes the design pattern known as Model-View-Controller (MVC). The 
architecture of this system is composed of three distinct layers, which are as follows: 
1. Model: Represents the application's core business logic and handles data operations, such as 
inquiry management, user records, and agency assignments. 
 
2. View: Manages the user interface, allowing users such as administrators, public users, and 
agencies to interact with the system and view information relevant to their roles. 
 
3. Controller: Acts as an intermediary between the View and Model, processing user inputs, 
triggering data operations, and determining appropriate views to render. 
 
 
The MVC pattern’s modular structure makes the system highly maintainable and scalable. 
Modifications to one component such as redesigning the user interface in the View or updating 
the database schema in the Model can be implemented with minimal impact on the other 
components. Furthermore, this pattern is well-suited for object oriented programming, enabling 
the use of encapsulation, inheritance, and polymorphism to streamline functionality and 
encourage code reuse. This makes MVC an ideal architectural choice for the Public Inquiry 
Management System, where multiple modules and user roles must operate seamlessly and 
efficiently within a robust, well-organized framework. 
 
 
3.2. Component Overview 
 
The " AuthenticityHub" system adopts a modular architecture, segmenting its core functionality into 
independent yet interconnected modules. Each module is designed to handle a specific part of the 
system, ensuring streamlined operations and smooth interaction between components. The key 
modules developed are Manage User Module, Manage Inquiry Form Submission Module, Inquiry 
Assignment Module, and Inquiry Progress Tracking Module
